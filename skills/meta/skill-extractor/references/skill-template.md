## 填写规范

### frontmatter 字段

| 字段 | 规则 | 示例 |
|------|------|------|
| `name` | 小写 a-z 0-9 连字符，1-64 字符 | `llm-provider` |
| `layer` | 枚举值之一 | `core` |
| `category` | 见分类标准 | `reasoning` |
| `status` | 新生成的统一为 `unverified` | `unverified` |
| `description` | 一句话 + 触发场景 + 关键词 | 见下方 |

**description 写法**：
```
[一句话描述]。
当用户[触发场景]时触发。
关键词：[关键词1]、[关键词2]、[关键词3]。
```

### 正文要求

1. **功能说明**：2-3 句话，不啰嗦。让 AI 快速判断是否需要加载这个 skill。
2. **能力概览**：表格形式，一目了然。
3. **使用方法**：代码示例必须可以直接复制使用，不能有占位符。
4. **配置**：如有配置项，列出表格。
5. **问题排查**：至少列 2-3 个常见问题。
6. **依赖**：版本号写死，不写"最新版"。
7. **来源**：标注原项目和模块路径，便于溯源。

### 禁止项

- ❌ 个人路径（如 `D:\Users\xxx\...`）
- ❌ 个人设备名
- ❌ 密码、Token、密钥
- ❌ "应该可以"、"可能"等不确定描述
- ❌ 原项目的业务逻辑（只提取通用模式）
- ❌ Lorem Ipsum 或其他占位文本

## 代码提取规则

### 什么时候提取代码到 scripts/

- 有可独立运行的脚本（接受输入、产出输出）
- 有配置模板（用户需要修改后使用）
- 有示例代码（帮助理解如何使用）

### 代码提取原则

1. **通用化**：去掉原项目的业务逻辑，保留通用模式
2. **参数化**：硬编码的路径/URL 改为参数或配置
3. **可运行**：确保代码可以独立运行（如有依赖在 SKILL.md 中注明）
4. **有注释**：关键逻辑加注释说明

### 目录结构

```
skills/<layer>/<category>/<skill-name>/
├── SKILL.md
├── scripts/
│   ├── [主功能].py          # 或 .js / .ts / .sh
│   └── [辅助功能].py
└── references/
    ├── [详细文档].md         # 补充说明（如有）
    └── [配置示例].yml        # 配置示例（如有）
```

## 生成检查清单

生成 SKILL.md 后，逐项检查：

- [ ] frontmatter 包含 name/layer/category/status/description 五个字段
- [ ] name 符合命名规则（小写连字符，1-64 字符）
- [ ] layer 和 category 值合法（见分类标准）
- [ ] description 包含一句话描述 + 触发场景 + 关键词
- [ ] 正文包含"功能说明"和"使用方法"
- [ ] 代码示例可直接复制使用（无占位符）
- [ ] 不包含个人路径、密码、设备名
- [ ] 依赖版本号已写死
- [ ] 标注了来源项目和模块
