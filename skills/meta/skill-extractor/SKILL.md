# Skill Extractor

从源码项目中提取可复用的 skill，通过 4 阶段对话式工作流完成拆解。

本 skill 供 AI Agent（如 mimoCode）消费。Agent 加载本文件后，按以下流程执行。
详细参考文档在 `references/` 目录下，按需读取。

## 执行流程

### 阶段 1：项目扫描

**目标**：理解项目全貌，识别所有可提取的功能点。

**步骤**：

1. 用户提供项目路径后，先读取项目配置文件了解概况
2. 扫描源码目录结构，识别主要模块
3. 分析每个模块的功能，提取可复用的功能点
4. 按类型（tool/skill/agent）和层级（core/action/identity/meta）分类

**扫描什么**：详见 [`references/scan-checklist.md`](references/scan-checklist.md)，核心是：
- README / CHANGELOG → 项目意图和功能范围
- package.json / pyproject.toml / Cargo.toml → 依赖和技术栈
- 目录结构 → 模块划分和边界
- 入口文件 → 核心调用链路
- 配置文件 → 支持的功能和扩展点

**分类标准**：详见 [`references/classification-guide.md`](references/classification-guide.md)

**输出格式**（展示给用户）：

```markdown
# [项目名] 功能点清单

## 项目概述
- 名称：[项目名]
- 语言：[主要语言]
- 架构：[架构描述]
- 版本：[版本号]
- 包结构：[如有 monorepo 则列出各包]

## 功能点列表

| # | 功能点 | 类型 | 层级 | 来源包 | 说明 |
|---|--------|------|------|--------|------|
| 1 | [名称] | tool/skill/agent | core/action/identity/meta | [模块路径] | [一句话说明] |

## 推荐拆解顺序
1. [低难度高价值的功能点] — 理由：[为什么]
2. [中难度高价值的功能点] — 理由：[为什么]
3. [高难度高价值的功能点] — 理由：[为什么]
```

**阶段结束条件**：功能点清单展示给用户，等待用户选择。

---

### 阶段 2：用户选择（多轮对话）

**目标**：通过对话确认用户要拆解哪些功能点。

**步骤**：

1. 展示阶段 1 的功能点清单
2. 等待用户选择（用户可以说"全部"、"第 1 和第 3 个"、或指定名称）
3. 如果用户选择不明确，追问确认
4. 列出最终确认的清单，请用户确认后继续

**对话规范**：
- 用户可以说编号、名称、或描述性选择（"所有 meta 层的"）
- 如果用户的选择跨越多个层级，提示依赖关系
- 如果用户选择的功能点有依赖关系，告知并建议拆解顺序
- 确认前展示最终清单，让用户有机会修改

**输出**：确认后的功能点列表（编号 + 名称 + 类型 + 层级）

**阶段结束条件**：用户明确确认选择。

---

### 阶段 3：Skill 生成

**目标**：为每个确认的功能点生成标准 skill 目录。

**步骤**：

1. 为每个功能点创建 skill 目录：`skills/<layer>/<category>/<skill-name>/`
2. 读取源码，提取核心逻辑和可复用模式
3. 编写 SKILL.md（遵循 CreateYouAI 格式）
4. 提取关键代码到 `scripts/`（如有可执行脚本）
5. 提取参考文档到 `references/`（如有需要补充的细节）

**SKILL.md 生成模板**：详见 [`references/skill-template.md`](references/skill-template.md)

**目录结构**：
```
skills/<layer>/<category>/<skill-name>/
├── SKILL.md              # 必须：给 AI 的指令文档
├── scripts/              # 可选：可执行脚本
│   └── [功能名].py
└── references/           # 可选：参考文档
    └── [文档名].md
```

**命名规则**：
- 小写字母 + 连字符分隔：`tool-calling` ✓，`Tool_Calling` ✗
- 1-64 字符
- 不以连字符开头/结尾，无连续连字符

**生成顺序**：按推荐拆解顺序生成（低难度优先），每生成一个就告知用户进度。

**阶段结束条件**：所有选定的功能点都生成了 skill 目录。

---

### 阶段 4：组合验证

**目标**：验证生成的 skill 格式合规、逻辑可用。

**步骤**：

1. 运行格式验证脚本检查每个 SKILL.md 是否合规
2. 分析 skill 之间的依赖关系
3. 生成一个简单的组合验证清单
4. 输出到指定目录

**格式验证**：运行 `scripts/validate-skill.py` 检查：
- YAML frontmatter 字段完整（name/layer/category/status/description）
- name 符合命名规则
- layer 和 category 值合法
- 正文结构完整（功能说明 + 使用方法 + 依赖）

**依赖分析**：
- 列出 skill 之间的引用关系
- 标注是否有循环依赖
- 建议加载顺序

**验证清单**：
```markdown
## 验证清单
- [✓/✗] 每个 SKILL.md 格式合规
- [✓/✗] 每个 skill 可独立理解
- [✓/✗] 依赖关系清晰无循环
- [✓/✗] 脚本可执行（如有）
- [ ] 用户手动验证通过
```

**阶段结束条件**：验证清单展示给用户，标注哪些已通过、哪些需要用户手动验证。

## Agent 执行规范

### 核心约束

1. **只读分析**：阶段 1-2 只分析不修改源码
2. **用户确认**：阶段 2 必须等用户明确确认后才继续
3. **标准格式**：阶段 3 输出必须符合 CreateYouAI 格式
4. **可验证**：阶段 4 必须输出验证清单
5. **不臆测**：如果不确定某个功能点的分类，坦诚告知用户并给出建议

### 分析原则

- **功能边界清晰**：一个 skill 只做一件事
- **依赖最小化**：skill 之间尽量独立
- **可复用性**：skill 应该能跨项目使用
- **提取模式**：关注可复用的设计模式，而非具体实现细节
- **保持通用**：生成的 skill 不应包含原项目的业务逻辑

### 对话风格

- 简洁直接，不废话
- 用表格和列表展示结构化信息
- 代码块展示技术细节
- 有不确定的地方明确标注，不假装知道

## 使用示例

**用户**：帮我从 D:\AI\参考\CODES\pi-main 拆出 skill

**Agent**：
1. 读取项目配置文件，扫描目录结构
2. 输出功能点清单（4 个功能点：skill-system / llm-provider / tool-calling / extension-system）
3. 用户选择"全部"
4. 逐个生成 skill 目录
5. 运行验证，输出验证清单

**真实案例**：Pi 项目的完整拆解过程详见 [`references/pi-example.md`](references/pi-example.md)

## 注意事项

- 如果项目太大（>100 个源码文件），建议分批拆解，先从核心模块开始
- 优先拆解低难度高价值的功能点（独立性强、依赖少）
- 生成的 skill 标记为 `unverified`，用户验证后改为 `verified`
- 如果项目是 monorepo，按 package 逐个分析
- 如果功能点跨多个模块，在 SKILL.md 的"依赖"部分注明来源
